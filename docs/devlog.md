# 開発日誌

## 2026-06-24（Codex）

### 実施内容
- `hskk.html` の Chrome 組み込み AI `LanguageModel` 呼び出しで、出力言語指定を `ja` に固定した。
- HSKK 部屋を v1.5 仕様に更新し、自由回答から提案文の読み上げへつなぐ往復練習ループを追加した。
- 褒め・提案前・再促しの文言を、陸の声に合わせたテンプレートからランダム表示するようにした。
- 提案文には `pinyin-pro` を使って声調記号付きピンインを表示するようにした。
- Nano rephrase 経路を残し、先に `question.model` を表示してから、短いタイムアウト内に有効な中文1文が返った場合のみ提案文を差し替える方式にした。
- HSKK の録音時間を `ANSWER_WINDOW_MS = 7000` で明示管理し、1回目回答と2回目「言ってみる」の両方に残り時間バーを表示するようにした。

### 修正理由
- Chrome の `LanguageModel` は `zh` を出力言語として指定できず、公開版で `Unsupported LanguageModel API languages...aborted` が出ていたため。
- AI にフィードバック全文を作らせると語り口が崩れるため、陸の声はテンプレートで固定し、AI は自然な中文への言い換えだけに役割を絞るため。
- Nano が無応答・未準備でも練習体験が止まらないように、常に既存のお手本文へ即時フォールバックする必要があったため。

### 実装メモ
- `expectedOutputs:[{type:'text',languages:['ja']}]` を維持し、中国語文はプロンプト内の文字列として扱う。
- `availability()` は 2.5 秒、`create()` と `prompt()` は 3 秒でタイムアウトし、失敗時は `question.model` を使う。
- 2回目の音声認識は既存の `SpeechRecognition` フローを再利用し、厳密一致ではなく緩い一致または発話ありで成功扱いにする。
- `voiceLogs` には `phase:'answer'` / `phase:'practice'` を保存して、自由回答と提案文練習を区別できるようにした。
- 録音中は `recognition.continuous = true` を使い、`no-speech` は即失敗にせず、7秒ウィンドウ内で再待機する。7秒経過時にアプリ側から `stop()` して結果処理へ進める。

### 確認方法
- `node` で `hskk.html` 内の JavaScript を構文チェックし、`JS parse OK` を確認した。
- ローカル HTTP サーバー経由で `hskk.html` を開き、`data/hskk.json` の読み込みと1問目の描画を headless Chrome で確認した。

## 2026-06-16

### 修正内容
- `idx` を `luca_save` に含め、会話位置を localStorage に保存・復元するようにした。
- メッセージ、音声、選択肢、ロック解除、Day終了で会話位置が進んだタイミングに保存するようにした。
- `restartAll()` を「最初から」ボタンとして扱い、`dayIdx`、`idx`、`love`、`awarded` を初期化して保存状態と画面上の親密度表示をリセットするようにした。
- Day開始処理で常に `idx` を 0 に戻さず、リロード時は保存済みの会話位置から再開できるようにした。

### 修正理由
- 会話途中でリロードすると Day は復元できても会話位置が先頭へ戻る可能性があったため。
- 「最初から」ボタンが読み返しに近い挙動で、親密度や報酬履歴が残る状態になっていたため。
- 選択肢報酬の取得済み履歴は維持しながら、会話位置だけを追加保存して二重取りを防ぐため。

### 確認方法
- `node --check /tmp/luca-script.js` で `index.html` 内の JavaScript 構文を確認する。
- ブラウザで会話途中まで進めてリロードし、同じ Day の保存済み会話位置から再開することを確認する。
- 選択肢を選んだあとにリロードしても、同じ選択肢報酬が再加算されないことを確認する。
- 「最初から」を押して Day1 先頭に戻り、親密度表示が Lv.1 / 0% 相当に戻ることを確認する。

### 残課題
- 会話復帰時に過去の吹き出し履歴を再構築するか、現在位置から表示するかの体験設計は今後見直す余地がある。
- 選択肢の返信文そのものは保存していないため、必要に応じて選択ログ保存を検討する。

## 2026-06-18（Claude Code）

### 実施内容
- プロフィール顔画像（`.profile-face`）の白縁問題を修正した。
- ヘッダーアバター・吹き出し横アバター・通話画面・レベルアップ演出の4箇所も同様の白縁問題を統一修正した。

### 問題の原因
- 各顔画像ファイル（PNG）の端に薄い白い余白が含まれていた。
- `object-fit: cover` / `background-size: cover` では画像をそのまま円に収めるため、白い縁がそのまま円の中に見えていた。
- 前回コミット（`object-fit: contain`）は白縁を隠すために試みたが、逆に白背景全体が見える状態になっていた。

### 修正方法

**プロフィール画像（`<img>` タグ）：**
- ラッパー `<div class="profile-face-wrap">` を追加し `overflow:hidden` で円クリップを担当させた。
- `<img>` 側に `transform:scale(1.1)` を適用し、白縁を円の外へ押し出した。

**CSS背景画像を使う4要素（`.avatar` / `.row .mini` / `.callface` / `#lvlupImg`）：**
- 各要素に `overflow:hidden; position:relative; background:#1a1530` を設定。
- `::before` 疑似要素に `inset:-10%` で親より10%大きい領域を確保し、そこへ背景画像を配置。
- `overflow:hidden` が円形にクリップするため、白縁は見えなくなる。

### JSの変更
- `showLevelUp()` 内でヘッダーアバターの computed backgroundImage を読んで `#lvlupImg` に設定していた処理を削除。
- `#lvlupImg::before` が `var(--luca-face)` を直接参照するため、JS側の設定は不要になった。

### 確認方法
- プロフィールパネルを開いて顔画像の円に白い隙間がないことを確認する。
- チャット吹き出し横の小アバター・ヘッダーアバターに白縁がないことを確認する。
- 通話画面・レベルアップ演出でも同様に確認する。

## 2026-06-17（Claude Code）

### 実施内容
- `index.html` に Base64 で直接埋め込まれていた画像 9枚を `assets/images/` へ外部ファイルとして書き出した。
- `index.html` 内の `data:image/...;base64,...` 参照をすべて相対パス（`assets/images/ファイル名`）に置換した。

### 外部化した画像ファイル
| ファイル名 | 用途 | サイズ |
|---|---|---|
| `luca-face-default.png` | CSS変数 `--luca-face`（チャット画面アバター） | 176KB |
| `bg.jpg` | 背景画像 | 102KB |
| `luca-face-call.png` | 通話オーバーレイ画面のフェイス | 200KB |
| `luca-face-0.png` ～ `luca-face-5.png` | `LUCA_FACES` 配列（表情差分6種） | 各176〜200KB |

### 結果
- `index.html` のファイルサイズ：**2,148KB → 53KB**（約97%削減）
- 残存する `data:image` 参照：**0件**

### なぜ今回できたか
- Codex Web はバイナリファイルを GitHub API 経由でプッシュできない制約がある。
- Claude Code はローカルクローン上でバイナリファイルを直接書き出して `git push` できるため、同じ作業を問題なく完了できた。
