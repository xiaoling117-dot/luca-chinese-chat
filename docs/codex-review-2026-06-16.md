# Codex Review: Luca Chinese Chat

調査日: 2026-06-16
対象: 恋愛チャット風中国語学習ゲームの試作

## 調査結果

このリポジトリは、`index.html` 1ファイルだけで動く静的プロトタイプです。HTML、CSS、会話データ、状態管理、UIイベント、音声読み上げ、画像アセット相当のBase64データがすべて同居しています。

コード本体は変更せず、調査時点では作業ツリーはクリーンでした。

## プロジェクト構成

### 全体構成

- アプリ本体は `index.html` のみです。
- `<head>` 内にviewport、タイトル、CSS変数、全スタイルが定義されています。
- `:root` には配色、吹き出し色、アクセント色、キャラクター顔画像のCSS変数が定義されています。
- `.phone` コンテナがスマホ風のアプリ画面全体を表します。
- `.phone` 内には、ヘッダー、親密度バー、チャット欄、選択肢・入力欄ドック、レベルアップ演出、単語帳パネル、プレゼントパネル、通話オーバーレイがあります。

### UIレイヤー

- ヘッダーには戻るリンク、アバター、オンライン状態、通話ボタン、親密度表示があります。
- 親密度ゲージは `.lovebar` で表示されています。
- チャット本文は `#chat`、選択肢は `#choices`、下部ショートカットは `#scVocab` と `#scGift` です。
- 疑似入力欄はdisabledな `<input>` で、「自由入力＝2回目（AI）で実装予定…」というplaceholderが入っています。
- 単語帳は `#vocabPanel` 内のフラッシュカードUIとして実装されています。
- プレゼント機能は `#giftPanel` と `#giftGrid` で構成されています。
- `+` メニューは `#plusMenu` に単語帳・プレゼントへの導線を持っています。
- 通話演出は `#callOverlay` に着信・応答・字幕・再生・次へ・終了ボタンをまとめています。

### データ構造

- 会話本編は `DAYS` 配列に日別で定義されています。
- 各Dayは `label`、`vocab`、`story` を持ちます。
- `story` には `msg`、`choice`、`voice`、`locked`、`end` などのノードが並びます。
- 選択肢には中国語、ピンイン、日本語訳、親密度加算、返信メッセージが含まれます。
- Day 5には「課金導線の演出」として `locked` ノードがありますが、実際はジョークとして解除できる構成です。
- 単語帳は各Dayの `vocab` を集約して作られます。
- 通話セリフは `CALL` 配列として本編とは別に定義されています。
- プレゼントは `GIFTS` 配列で、絵文字、名称、中国語、親密度加算、1日上限を持っています。

### 状態管理

- ゲーム状態は `dayIdx`、`idx`、`love`、`awarded` で管理されています。
- `save()` / `load()` は `localStorage` の `luca_save` に親密度、日付インデックス、付与済み選択肢キーを保存・復元します。
- プレゼントの1日制限も `localStorage` に保存されます。
- プレゼントの日次キーは `new Date().toISOString().slice(0,10)` で作られているため、UTC日付ベースです。

### 描画・進行ロジック

- `run()` がストーリー進行の中心です。
- 現在ノードの `type` に応じて、メッセージ表示、音声バブル表示、選択肢表示、ロック表示、終了カード表示を切り替えます。
- 選択肢クリック時は `pick()` がユーザー発話を追加し、親密度を加算し、必要なら返信を挟んで進行します。
- `ending()` はその日の復習単語と翌日/最初からボタンを表示します。
- `startDay()` はチャット欄と選択肢を初期化し、その日のタイムスタンプを表示してから `run()` を開始します。

## Safariで起きそうな問題

### 1. `100vh` によるアドレスバー・下部バー問題

`.phone` が `height:100vh` で固定されています。iOS Safariではアドレスバーや下部ツールバーの表示/非表示で `100vh` の解釈が変わり、下部の選択肢・入力欄ドックが隠れる、または画面下にはみ出す可能性があります。

特にこのアプリは下部に `.dock`、選択肢、ショートカット、入力欄が集中しているため、Safariの動的viewportの影響を受けやすいです。

改善案:

- `height: 100dvh` を検討する。
- `min-height: -webkit-fill-available` を併用する。
- `padding-bottom: env(safe-area-inset-bottom)` を入れる。

### 2. Safe Area未対応

iPhoneのホームインジケータ領域に対する `env(safe-area-inset-bottom)` が使われていません。`.dock` は下端に配置されるため、iPhone Safariでは入力欄・ショートカットがホームインジケータに近すぎる、または重なる可能性があります。

改善案:

```css
.dock {
  padding-bottom: calc(14px + env(safe-area-inset-bottom));
}
```

### 3. `backdrop-filter` の互換性

ヘッダー、ドック、パネル、レベルアップ演出で `backdrop-filter` を使っています。Safari自体は対応していますが、古いSafariや一部WebViewでは `-webkit-backdrop-filter` が必要になる場合があります。

改善案:

```css
backdrop-filter: blur(8px);
-webkit-backdrop-filter: blur(8px);
```

### 4. Web Speech APIの音声ロード問題

音声再生は `speechSynthesis` と `SpeechSynthesisUtterance` を使っています。

Safariでは中国語音声が端末に入っていない場合、`zh-CN` を指定しても日本語/英語音声で読まれる、無音になる、または初回の `getVoices()` が空になることがあります。

現在 `speechSynthesis.onvoiceschanged=function(){}` は空実装なので、声一覧の読み込み後にUIを更新する処理や再選択はありません。

改善案:

- voicesのロード完了を待つ。
- 利用可能な中国語音声がない場合の表示を入れる。
- 読み上げ中のボタン連打を抑制する。
- `SpeechSynthesisUtterance.onend` / `onerror` を使って再生状態を管理する。

### 5. Base64巨大CSS/JSによるSafariのメモリ・初期描画負荷

`--luca-face` と `.phone` 背景にBase64画像が埋め込まれています。さらに `LUCA_FACES` に巨大なBase64画像配列が定義されています。

このファイルは約2.2MBの単一HTMLで、Safariの初期パース・スタイル計算・メモリ使用量が増えやすいです。特にiPhoneの低メモリ環境では、初回表示やレベルアップ時の背景差し替えが重くなる可能性があります。

改善案:

- 画像は外部ファイル化する。
- 必要に応じてプリロード/遅延読み込みする。
- CSS変数に巨大Base64を直接入れない。

### 6. `user-scalable=no` によるアクセシビリティ問題

viewportで `maximum-scale=1.0, user-scalable=no` が指定されています。

Safariでは文字が小さいUIや中国語・ピンイン確認時にズームできないため、アクセシビリティ上の問題になります。

改善案:

- `maximum-scale=1.0, user-scalable=no` を外す。
- ユーザーがズームできるようにする。

### 7. UTC日付でプレゼント制限が切り替わる

プレゼントの日次キーが `new Date().toISOString().slice(0,10)` で作られています。

これはUTC日付なので、日本時間やユーザーの現地時間と日付切り替わりがズレます。Safari固有ではありませんが、モバイルSafariで日常的に使う場合、「まだ今日なのにリセットされた/日付が変わったのにリセットされない」と感じる可能性があります。

改善案:

- ローカル日付ベースで `YYYY-MM-DD` を作る。
- もしくはゲーム内の日付リセット仕様を明示する。

## モバイル表示の問題

### 1. 画面下部が詰まりやすい

下部ドックには選択肢、ショートカット、入力欄が縦に積まれます。

選択肢が3つある場面では `.choices` に複数ボタンが入り、さらにショートカットバーと入力欄が続くため、小さいiPhone SE系の高さではチャット本文がかなり狭くなります。

改善案:

- 小画面ではショートカットバーを折りたたむ。
- 未実装の入力欄を非表示または小さくする。
- 選択肢をコンパクト化する。

### 2. 無効な入力欄が画面スペースを消費している

入力欄は「自由入力＝2回目（AI）で実装予定…」というplaceholder付きでdisabledになっています。

プロトタイプとして意図は分かりますが、モバイルでは貴重な下部スペースを占有します。さらにdisabledなのでタップしても何もできず、ユーザーが「入力できないバグ」と誤解する可能性があります。

改善案:

- 未実装なら小さな説明ラベルにする。
- 初回だけ「自由入力は次回実装」とトースト表示する。
- 今回の試作では入力欄を非表示にすることも検討する。

### 3. `.phone` の最大高さと中央寄せが実機で不自然になる可能性

PC幅では `.phone` が `height:92vh`、角丸付きのスマホモックになります。一方、モバイル幅では `height:100vh` の全画面になります。

この切り替え自体は良いですが、実機Safariの動的バーと組み合わさると、下部だけでなく上部ヘッダーも微妙に押し出されることがあります。

改善案:

- `100dvh` とSafe Area対応をセットで入れる。
- 端末の縦横・小画面高さのメディアクエリを追加する。

### 4. 選択肢内の日本語訳が横並びで詰まる

選択肢ボタンでは中国語本文の後ろに `.cgloss` の日本語訳が `margin-left` 付きで同じ行に入ります。

短文なら問題ありませんが、長い中国語や日本語訳では横幅が足りず、読みにくい折り返しになります。

改善案:

- モバイルでは中国語と日本語訳をブロック表示にする。
- 2行構成にして読みやすくする。

### 5. タップ対象の意味が分かりにくい

吹き出しはタップで訳を開く仕様で、右下に小さく「訳」と出ます。

恋愛チャットとしては自然ですが、初見ユーザーには「翻訳が隠れている」ことが分かりにくい可能性があります。

改善案:

- 初回だけ「吹き出しをタップすると訳が見られます」とチップ表示する。
- 「訳」ラベルを少し大きくする。
- 開閉可能なUIであることを視覚的に示す。

### 6. 通話オーバーレイの下部ボタンがSafe Area非対応

通話ボタン群は `bottom:46px` で固定配置されています。

iPhoneのホームインジケータやSafari下部バーの状態によっては、ボタンが押しづらくなる可能性があります。

改善案:

```css
.callbtns {
  bottom: calc(46px + env(safe-area-inset-bottom));
}
```

### 7. 単語帳・プレゼントパネルが全画面固定で閉じづらい可能性

パネルは `position:absolute; inset:0;` で全画面を覆います。閉じるボタンは右上にありますが、モバイル片手操作では遠くなりやすいです。

改善案:

- 下部にも「閉じる」ボタンを置く。
- スワイプダウンで閉じる。
- 背景タップで閉じる。
- パネルヘッダーをSafe Area対応にする。

## 改善すべき箇所

### ファイル分割

現在は `index.html` 1ファイルに、HTML、CSS、ストーリーデータ、ロジック、Base64画像が全部入っています。

短期の試作としては速いですが、今後会話量やキャラ画像が増えると保守がかなり難しくなります。

おすすめ構成:

```text
index.html
src/
  styles.css
  data/days.js
  data/call.js
  data/gifts.js
  app.js
assets/
  luca/
    face-lv1.png
    face-lv2.png
    ...
    bg.jpg
```

### `innerHTML` の使用箇所を減らす

ストーリーデータ由来の文字列を `innerHTML` に連結している箇所があります。

現状はデータが同一ファイル内で固定なので大きな脆弱性には直結しにくいですが、将来AI生成文・外部JSON・管理画面入力に変わるとXSSリスクになります。

改善案:

- `textContent` と `createElement()` ベースに寄せる。
- どうしてもHTMLを使う場合はサニタイズする。
- ストーリーデータを信頼境界として扱う。

### Safari/iPhone向けviewport対応

`height:100vh`、Safe Area未対応、`user-scalable=no` はモバイル体験の主要リスクです。

改善案:

```css
html, body {
  min-height: 100%;
}

.phone {
  height: 100dvh;
}

.dock {
  padding-bottom: calc(14px + env(safe-area-inset-bottom));
}
```

あわせてviewportの `user-scalable=no` は削除推奨です。

### 進行状態の保存粒度

現在保存しているのは `love`、`dayIdx`、`awarded` で、会話中の `idx` は保存されていません。

そのため、途中でリロードするとその日の最初から再生される一方、親密度付与済みキーは残っているため、同じ選択肢を選んでも親密度が増えない状態になります。これはユーザーには不自然に見える可能性があります。

改善案:

- `idx` も保存する。
- あるいは日単位でしか保存しない仕様にして「途中復帰なし」と明示する。

### `restartAll()` が親密度や付与済み状態をリセットしない

`restartAll()` は `dayIdx=0; idx=0; startDay();` のみで、`love` や `awarded` を初期化していません。

「最初から」と表示されるボタンなのに親密度が残るため、ゲームとしては違和感があります。

改善案:

- 本当に最初からなら `love=0; awarded=new Set(); save(); updateLove(false);` まで行う。
- 読み返し機能なら文言を「Day 1から読み返す」に変える。

### レベル計算の仕様確認

親密度レベルは `Math.floor(v/17)+1` で最大Lv.6です。

`v=100` でも最大Lv.6に収まるため動作は問題ありません。ただし、17刻みはやや分かりにくいので、レベルテーブル化すると演出調整が楽になります。

### 通話の親密度付与が1回だけ

通話終了時に `awarded` に `'call'` がなければ親密度+8されます。

これは全期間で1回だけの扱いです。毎日1回の通話報酬にしたい場合は `call_${dayIdx}` や日付キーにする必要があります。

### プレゼント親密度が通常セーブに保存されない

プレゼント送信時は `love` を増やして `updateLove(true)` していますが、通常の `save()` は呼ばれていません。

`sent` は保存されるため、リロード後は「プレゼントは使用済みなのに親密度が戻る」可能性があります。これはかなり気づきやすい不整合です。

改善案:

- プレゼント送信時に `save()` も呼ぶ。

### 声再生ボタンの状態管理

音声バブルの再生では、押した時に `setTimeout` で見た目を戻しています。

実際の読み上げ完了イベントとは連動していないため、Safariや端末設定によっては、音声が終わっていないのに再生表示が戻る、または無音でも再生中表示になる可能性があります。

改善案:

- `SpeechSynthesisUtterance.onend` / `onerror` を使って状態を戻す。

### CSSのPC/モバイル調整が少ない

メディアクエリは実質 `.phone` の角丸/高さ調整のみです。

モバイル向けには、小画面高さ、横幅320px前後、ノッチ付き端末、横向きなどを追加で考慮したいです。

### アクセシビリティ

- 通話ボタンには `aria-label` がありますが、`+` ボタンは `div` で、キーボード操作やスクリーンリーダー上の意味が弱いです。
- パネルを開いた時のフォーカストラップやEscapeキーで閉じる処理はありません。
- 吹き出しの「訳」開閉もボタンではなくdivクリックなので、支援技術では操作可能性が伝わりにくいです。

改善案:

- インタラクティブ要素は `button` 化する。
- `aria-expanded`、`aria-controls`、フォーカス管理を入れる。
- Escapeキーや背景クリックで閉じる動線を入れる。

### コンテンツ制作のしやすさ

ストーリー配列は読みやすく書かれていますが、日数が増えるほど1ファイル内のスクロールが大変になります。

改善案:

- DayごとにJSON/JSファイルを分ける。
- 簡単なスキーマチェックを入れる。
- セリフ追加時の構文ミスを検出できるようにする。

### テストしやすさ

現状はビルドツールやテストがなく、ブラウザで開いて確認する前提です。

静的プロトタイプとしては問題ありませんが、`DAYS` の構文、必須キー、空の `vocab`、不正な `type` などを検出する仕組みがありません。

改善案:

- まずはNodeで `DAYS` を検証する簡単なスクリプトを作る。
- HTMLからJSを抽出して構文チェックするコマンドを用意する。
- 将来的にはPlaywright等で主要画面の表示確認を自動化する。

## 優先度高の修正項目

1. Safari/iPhone向けに `100dvh` とSafe Area対応を入れる。
2. viewportから `user-scalable=no` を外す。
3. プレゼント送信時に `save()` を呼び、親密度がリロードで消えないようにする。
4. `restartAll()` の仕様を決め、完全リセットまたは読み返しとして文言・状態管理を揃える。
5. 会話途中のリロード挙動を整理し、`idx` を保存するか、日単位復帰に限定する。
6. Web Speech APIの音声ロードと再生状態管理を改善する。
7. 巨大Base64画像を外部アセット化し、単一HTMLの肥大化を解消する。
8. `innerHTML` 連結を減らし、将来の外部データ化に備えてXSSリスクを下げる。

## 特に気になった不具合候補

1. プレゼントで増えた親密度がリロードで消える可能性があります。
2. 「最初から」が完全リセットではありません。
3. 途中リロード時の復帰が中途半端です。
4. iOS Safariで下部UIが隠れる可能性が高いです。
5. Web Speech APIはSafariで不安定になりがちです。
